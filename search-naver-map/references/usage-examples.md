# 사용 예시

같은 질문이라도 사용자가 이미 Place ID나 예약 URL을 알고 있는지에 따라 필요한 명령은 달라집니다.

## 날짜와 인원이 맞는 숙소 찾기

> 제주 게스트하우스 중 7월 20일부터 22일까지 남자 2명이 묵을 수 있고, 서귀포가 아니며, 조회 시점에 예약 가능으로 표시된 객실이 있는 곳을 찾아줘.

확인할 정보:

- 검색된 장소의 지역과 업종
- 공개 Booking 링크가 있는지
- 날짜별 객실, 인원 수용 정보, 가격과 재고
- 확인하지 못한 후보와 제외 이유

예약 정보가 없거나 수용 인원을 판단할 근거가 부족하면 `is_available: null`로 남깁니다. 조회 결과는 예약 신청이 아니며 이후 재고를 보장하지 않습니다.

## 메뉴가 확인되는 음식점 찾기

> 성수에서 떡볶이 메뉴가 공개된 매장을 찾아서 영업시간과 방문자 리뷰를 비교해줘.

확인할 정보:

- 지도 검색 결과의 장소 후보
- Place 상세에 공개된 메뉴와 가격
- 영업시간과 현재 영업 상태
- 공개 리뷰 본문과 태그

검색어에 떡볶이가 들어갔다는 이유만으로 판매 메뉴를 단정하지 않습니다. 메뉴나 상세 설명에서 확인되지 않으면 근거가 없다고 표시합니다.

구조화 메뉴가 비어 있거나 요청한 메뉴·가격을 답하기에 부족하면 `detail --view full`에서 공개 Place 이미지를 가져와 [메뉴 이미지 fallback](menu-image-fallback.md)을 적용합니다. 메뉴판 글자가 판독될 때만 이미지 근거로 메뉴명과 가격을 제시합니다. 음식 사진만 보이거나 글자가 흐리면 판매 메뉴로 확정하지 않습니다.

## 매장의 검색 노출 확인하기

> 잠실 꽃집 검색에서 우리 매장이 어디에 보이는지 확인하고, 주변 매장의 영업시간·리뷰·예약 노출과 비교해줘.

확인할 정보:

- 조회된 지도 검색 화면에서의 현재 위치
- 장소별 공개 상세 정보와 리뷰 수
- 공개 Booking 링크와 상품·시간대 노출

`search`의 위치는 조회된 한 화면 기준입니다. 전체 검색 결과의 절대 순위나 다른 기기·지역·개인화 환경의 순위를 뜻하지 않습니다.

변화를 계속 추적하려면 외부 작업이 `--output` JSON을 날짜별로 보관하고 비교해야 합니다. 이 저장소에는 스케줄러나 이력 데이터베이스가 없습니다.

## 리뷰 표현으로 이용 상황 살펴보기

> 직장인이 많이 찾는 식당을 공개 리뷰 근거로 찾아줘.

리뷰 본문에 회식, 점심시간, 퇴근 같은 표현이 공개돼 있다면 관련 표현을 근거로 제시할 수 있습니다. 다만 리뷰 작성자의 직업과 직장인 이용 비율은 확인할 수 없습니다.

직업 정보가 없으면 닉네임이나 방문 시간만으로 직장인이라고 단정하지 않고 그 한계를 명시합니다.

## 특정 시간에 예약 가능한 식당 찾기

> 이번 금요일 오후 7시부터 8시 사이에 두 명이 예약 가능한 식당을 찾아줘.

확인할 정보:

- 공개 Booking을 제공하는 장소
- 요청 날짜의 시간대와 잔여 상태
- 두 명을 받을 수 있는지 확인할 수 있는 정보

공개 응답에 시간대나 수용 인원이 빠져 있으면 예약 가능으로 추정하지 않습니다. 예약 신청과 결제는 사용자가 네이버에서 직접 진행해야 합니다.

## 오프라인으로 결과 형식 재현하기

개발과 테스트에서는 저장된 응답을 사용해 네트워크 없이 같은 파서와 JSON 형식을 확인할 수 있습니다.

```bash
bin/naver-place search \
  --query "샘플 장소" \
  --html tests/fixtures/map/search-success.html
```

저장된 응답의 기록 시각이 없다면 결과의 `fetched_at`은 `unknown`입니다. 테스트를 실행한 시간이 최신 장소·예약 정보를 뜻하지는 않습니다.
